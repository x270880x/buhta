# CLAUDE.md — Бухта, кадастровая карта

Контекст проекта. Подробности — в `ONBOARDING.md` и `PROJECT_LOG.md` (читать при старте сессии).

## Суть
Интерактивная кадастровая карта **с. Бухта** — Leaflet-одностраничник, без бэкенда.
⚠️ **с. Бухта = Генический район, Херсонская область.** НЕ Кирилловка, НЕ Запорожье — не возвращать старые укр. привязки.

## Структура
- `index.html` — вся карта (Leaflet 1.9.4 + leaflet-rotate, bearing 60°)
- `councils/2320355400.json` — метаданные участков
- `parcels/2320355400.geojson` — геометрия (522 полигона)
- `ONBOARDING.md` / `PROJECT_LOG.md` — полный контекст и история

## Поля участка (councils JSON)
`c` кадастр укр · `a` га · `o` владелец · `ot` форма собств. · `zn` массив ·
`fias` адрес РФ · `cadrf` кадастр участка РФ ·
`egrn` («да») · `post` постановление · `ct` ВРИ · `rr` («да» = (пере)регистрация в Росреестре по Херсонской обл. → зелёный флажок) ·
`zaddr` старый запорожский адрес (из докс-таблицы ФИАС; НЕ выводить в UI — хранится скрыто по просьбе пользователя) ·
`mfc`/`bmfc` («да» = участок/дом подан на регистрацию в МФЦ, ждёт; после регистрации ставить rr/brr и снимать mfc/bmfc).

**`blds`** — массив зданий участка: `[{ lit, area, cad, rr }]`. Дома А, Б, В — три отдельных элемента (и три отдельных дома в статистике).
`lit` литера/номер · `area` площадь · `cad` кадастр здания РФ · `rr` («да» = сам ДОМ перерегистрирован в Росреестре ХО; `rr` участка на дом НЕ распространяется).
В попапе зданиe с `rr` рисуется зелёным (#2ecc71) + значок «🟢 РР ХО».
Старые поля `building` / `bcadrf` / `brr` заменены на `blds` (v5.2); `buildingsBlock()` в index.html ещё читает `building`/`bcadrf` как фолбэк.
Топ-левел `unbound` в JSON — дома без привязки к укр. участку (Набережная 46 А/Б/В), та же форма (`lit`/`cad`/`area`/`rr`); видны только в статистике, на карту попадут после привязки.

## Кнопка «ИНФО» (заменила «Регистрации»)
Единый реестр всех объектов с адресом ФИАС ХО: участки + их дома (вложенно) + `unbound`. Колонки: адрес · размер · владелец · кадастр РФ + укр.
Цвет строки (левая полоса, точка, кадастр РФ):
- 🟢 `#2ecc71` — `rr` есть: адрес сменён / право зарегистрировано в Росреестре ХО;
- 🟡 `#e8c547` — `dst:"stop"` (приостановка), причина в `dwhy`;
- 🔴 `#e23b3b` — `dst:"deny"` (отказ), причина в `dwhy`;
- ⚪ `#7a7a7a` — документа нет; подпись = `note` из `registrations.json → pending`, иначе «постановление № … — в Росреестре адрес ещё не сменён».

⚠️ Поле `st` в участках **занято исходными данными ДЗК** (значение «р») — под статус документа заведены `dst` + `dwhy`.
Фильтр-чипы по статусу, поиск по адресу/владельцу/кадастру (слова по «И», работает и в неправильной раскладке), клик по строке → переход к участку.

## registrations.json
Номера госрегистрации права из выписок ЕГРН, сгруппированы по владельцу (`owners[]`) + `pending[]` (право ещё не зарегистрировано).
Подтягивается в «ИНФО»: `owners[].items[].reg` → строка «рег. права …», `pending[].note` → причина у ⚪.
**Номер КУВД в выписках не печатается** — используется номер записи о госрегистрации права (раздел 2, п. 2.1).

## Правила работы
1. Обработка уведомлений: извлечь кадастр (укр `2320355400:…` / РФ `96:01:…`) + адрес (улица+ЗУ N) + здание + № постановления. Матч по `p.c` или `p.cadrf`. Суффикс бывает в кв. 12:001 и 11:003 — разрешать по владельцу/адресу.
2. PDF-сканы: `pymupdf` → PNG → читать визуально. **Проверять крупным планом** — беглое чтение даёт ошибки.
3. Владельца менять только по явной просьбе.
4. После правки: `commit + push`, **бампить BUILD_VERSION в index.html**, дать ссылку `?v=<sha>`.
5. Коммиты/код/пути — на английском; общение — по-русски.
6. Жукова (если появится) → фактическая хозяйка Моршнева Н.В.
